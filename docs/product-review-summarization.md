# Ревью суммаризации: статьи + комментарии

> Дата: 14 июля 2026. Взгляд продуктолога (AI-суммаризация, рассылки, блоги) на ядро продукта — генерацию саммари.
> Основано на: `pipeline/summarize.ts`, `utils/summary-guard.ts`, `utils/summary-heuristics.ts`, `utils/html-to-md.ts`, `eval/*`, реальные продовые саммари (132 поста + 150 комментов за июнь–июль 2026), `failed_summaries_analysis.txt`.
> Дополняет общее `product-review.md` от 13 июля — здесь только про качество и формат саммари.

## Что уже сильно

Инженерная обвязка выше среднего для инди-продукта: идемпотентность по `inputHash`, cooldown, приоритизация кандидатов, цепочка fallback-моделей, три попытки с ужесточением промпта, эвристики на отказы/повторы/prompt-leak, LLM-guard, оффлайн-бенч с judge (accuracy/completeness/faithfulness/format). Такой каркас есть не у каждой коммерческой рассылки. Проблемы ниже — не «переделать всё», а «дожать конкретные звенья».

## Главные находки (с доказательствами из прода)

### 1. Извлечение контента — самое слабое звено всей цепочки

`utils/html-to-md.ts` — это turndown **всей страницы** (удаляются только `script`/`style`), а `buildPostPrompt` берёт **первые** `ARTICLE_SLICE_CHARS=6000` символов. На современных сайтах первые 6к символов — это навигация, куки-баннер, футер.

Доказательства:
- Пост 48845049 («18 Words»): саммари добросовестно пересказывает **интерфейс** — кнопки «Практиковаться снова», сообщение об ошибке «Something went wrong. Please try again», дважды ссылку на Zanagrams. Guard пропустил с confidence 0.9 — потому что саммари *верно* пересказывает мусорный вход.
- `failed_summaries_analysis.txt`: все содержательные отказы моделей — это футер YouTube, навигация Yahoo, меню GitHub вместо статьи.
- Хвостовое следствие: у части постов саммари вообще нет (напр. 48849066 — есть теги и комменты, поста нет).

Guard по построению не ловит этот класс: он проверяет «соответствует ли саммари тексту», а не «является ли текст статьёй».

**Что делать.** Readability-экстракция перед turndown (`@mozilla/readability` + `jsdom` — jsdom уже в зависимостях). Затем дешёвый детектор мусора ДО трат LLM-квоты: доля ссылок/списков, длина связного текста, повторяемость строк. Если экстракт мусорный — не жечь токены на 3 попытки × 3 модели, а пометить `degraded: no-article` и жить только с саммари комментов. Плюс: брать не только первые 6к символов — для длинных статей терять заключение больно; дешёвая схема «начало 4к + конец 2к».

### 2. Языковая чистота: систематические английские вкрапления в русском тексте

Продовая метрика: у **54 из 132** свежих RU-саммари (41%) >8% латинских токенов. Часть — легитимные имена собственные, но выборочное чтение подтверждает именно смешение языков в связном тексте:

- «такие меры создают **precedents**» (48843923);
- «система очков **lets you compare results globally**» (48845049);
- «из‑за спешного голосования и требования абсолютного большинства **для от rejection** он прошёл» (комменты 48843923).

Причина: primary-модель `nemotron-3-nano-30b-a3b:free` — маленькая, ей одновременно поручают компрессию **и** перевод en→ru. Ни эвристики, ни guard, ни judge (`eval/score-models.ts` — рубрика без language-измерения) чистоту языка не проверяют. Для RU-first продукта это витрина: дайджест с ломаным русским считывается как спам-переводчик, доверие не возвращается.

**Что делать (по нарастанию):**
1. Эвристика в retry-цикл: доля кириллицы в буквах < ~85% (вне кода/имён) → следующая попытка. Инфраструктура ретраев уже есть, не хватает триггера.
2. Эскалация модели при content-reject: сейчас после guard/heuristic-отказа ретрай бьёт в ту же primary (fallback включается только на HTTP-ошибках). Вторая-третья попытка должна идти сразу на `llama-3.3-70b` — иначе трижды повторяем системную ошибку маленькой модели.
3. Эксперимент через уже существующий бенч: «саммари на EN (родной язык источника и модели) → отдельный шаг перевода на RU» против прямого RU. На маленьких моделях двухшаговка часто радикально чище. Бенч-харнесс готов, стоимость проверки — вечер.
4. Добавить `language_purity` в рубрику judge.

### 3. Guard проверяет не то и слепа к фактам

Кейс 48843923 (Chat Control, топовая тема): саммари называет активиста «**Эдвард** Брейер», guard в своих же reasons пишет «**Patrick** Breyer's concerns» — оба имени были у guard перед глазами, вердикт ok/0.9. Фактическая ошибка в самой видимой новости дня.

Ещё: guard сверяет саммари с обрезком в `POST_GUARD_ARTICLE_MAX_CHARS=4000`, а саммари делается из 6000 — последняя треть контекста для guard невидима, галлюцинации «из хвоста» непроверяемы в принципе.

**Что делать.** Расширить JSON guard'а тремя полями: `is_source_article` (это статья или navigation/boilerplate?), `entity_mismatches[]` (имена/числа/даты из саммари, которых нет в тексте — прямая инструкция сравнить), и поднять guard-окно до `ARTICLE_SLICE_CHARS`. Groq llama-4-scout со structured outputs это уже умеет, стоимость примерно та же.

### 4. Комментарии — заявленная главная ценность продукта — получают меньше всего заботы

Асимметрия поста и комментов:

| | Пост | Комменты |
|---|---|---|
| Попыток | 3 (initial, strict-1, strict-2) | 1 |
| Эвристики | да | **нет** |
| Guard | да | **нет** |
| Контекст в промпте | статья | **даже без заголовка поста** |

Конкретика:
- `buildCommentsPrompt` не передаёт **заголовок статьи**: модель суммаризирует спор, не зная его темы. Дешевейшее из всех улучшений — одна строка.
- Каждый коммент обрезается до **400 символов** — лучшие комменты HN (развёрнутый опыт инженера) как раз длинные, обрезка убивает их первыми.
- Плоский формат `@user [d2] text`: структура тредов (кто кому возражает) потеряна — а просили ведь пересказ *спора*.
- RU-формат жёстче EN без причины: EN — «5–7 буллетов», RU — «3–5». Прод-факт: 89% саммари — ровно 4 буллета, медиана 70 слов, и это на треды по 150–300 комментов.
- 40 комментов / depth≤2 — для больших тредов покрытие ~13%, споры уровня 3+ не существуют.
- В Telegram `commentsSummary` вообще выкинут (`buildTelegramItemFromStory` → `commentsSummary: undefined`): главный канал дистрибуции доставляет продукт без главной фишки.

**Что делать.** (а) Заголовок + TL;DR статьи в промпт комментов. (б) Валидация как у постов — минимум эвристики + языковой гейт + 1 ретрай. (в) Лимит на коммент 400→900 для топ-веток. (г) Структурированный выход вместо плоских буллетов: `{consensus, споры[{позиция_А, позиция_Б}], лучшая_цитата{автор, текст}, практические_советы[]}`. Цитата с атрибуцией и ссылкой — самый расшариваемый артефакт в жанре рассылок (Money Stuff, TLDR живут на этом). (д) В TG добавить блок «💬 О чём спорят» на 1–2 строки.

### 5. Один промпт и одна длина для всего

Промпт один: «~170 слов, два абзаца, главная идея + яркие факты» — и для релиза bat, и для научного PDF, и для YouTube-доклада, и для Show HN-игры. Прод-факт: p50=168, p90=178 слов — модель **всегда** пишет ~170 слов, независимо от веса источника. Тонкий контент надувается повторами (Zanagrams дважды в 48845049), плотный — ужимается с потерями.

При этом сигналы типа уже есть в пайплайне (ветки PDF/YouTube, URL-паттерны, префикс «Show HN:»), но в промпт не пробрасываются. Отдельная потеря: `normalizeStory` выбрасывает `item.text` — авторский питч Show HN/Ask HN не доходит до суммаризации вовсе.

**Что делать.** Лёгкий роутинг типа контента (release / paper / show-hn / video / blog) с одной-двумя типо-специфичными строками в системном промпте: релиз — «что изменилось и стоит ли обновляться», paper — «проблема→метод→результат→ограничения», Show HN — «что делает, как устроено, кому нужно» (+ вернуть `item.text` в пайплайн). И заменить фикс «~170 слов» на «столько, сколько заслуживает источник, максимум 170; для тонких источников 40–60 слов — норма».

### 6. Один текст на все поверхности

`postSummary` без изменений идёт в карточку ленты, страницу поста и Telegram. Ремесло рассылок: у каждой поверхности своя длина. Лента — хук в одну строку (решение «читать/скип» за взгляд), TG — 2–4 предложения + почему важно, страница — полный текст. Общее ревью уже предлагало TL;DR; здесь уточнение реализации: **не отдельным вызовом**, а structured output в том же вызове суммаризации — `{tldr: ≤15 слов, summary, why_care: 1 предложение}`. Нулевая доп. стоимость токенов, схемы через Groq уже освоены. Это же поле — готовая тема письма для будущего email-дайджеста и первая строка TG-поста.

### 7. Eval существует, но петля не замкнута

Бенч с judge — редкая роскошь, но: гоняется вручную и оффлайн; рубрика без языковой чистоты; саммари комментов не оцениваются вообще; продакшн не сэмплируется — поэтому 41% языкового смешения жил незамеченным. `failed_summaries_analysis.txt` — октябрь 2025, разовый ручной прогон девятимесячной давности.

**Что делать.** Еженедельный автопрогон judge по 15–20 случайным продовым саммари (посты + комменты) с трендом метрик — рядом с планируемой статистикой LLM-затрат (`plan-llm-usage-stats.md`). Любая смена модели — только через бенч-гейт. Плюс дешёвый пользовательский сигнал: 👍/👎 на карточке событием в GoatCounter — сейчас от читателей ноль обратной связи, а это единственный внешний детектор «саммари врёт».

### 8. Доверие: саммари не подписаны как ИИ

Нигде на карточке/странице нет пометки «саммари сгенерировано ИИ (модель, дата), возможны ошибки». Для продукта, чья валюта — доверие к пересказу, одна громкая галлюцинация без дисклеймера = потерянный читатель + скриншот в соцсетях. Модель и дата уже хранятся в JSON — вывести их в UI дёшево, а «Эдвард Брейер» показывает, что случай не гипотетический.

## Приоритеты

### Быстрые победы (дни, почти всё — обвязка, не LLM)

| # | Идея | Эффект | Стоимость |
|---|---|---|---|
| 1 | Readability + детектор мусорного экстракта до LLM | Убивает целый класс браков (все кейсы из failed-analysis), экономит квоту | Низкая (jsdom уже есть) |
| 2 | Языковой гейт (доля кириллицы) в retry-цикл + эскалация на 70B при content-reject | Режет главный видимый брак — 41% смешения | Низкая |
| 3 | Заголовок статьи в промпт комментов | Точность саммари споров | Тривиальная |
| 4 | Эвристики + ретрай для саммари комментов | Паритет контроля качества с постами | Низкая |
| 5 | RU-комменты: 5–7 буллетов как в EN | Полнота на больших тредах | Тривиальная |
| 6 | Блок «О чём спорят» в Telegram | Главная фишка продукта в главном канале | Низкая |
| 7 | Пометка «ИИ-саммари, модель, дата» в UI | Страховка доверия | Тривиальная |

### Среднесрочно (1–2 недели)

- Structured output `{tldr, summary, why_care}` одним вызовом → карточки, TG, будущий email.
- Апгрейд комментов: структура тредов в промпте, лимит 400→900 для топ-веток, выход `{consensus, споры, цитата}`.
- Guard v2: `is_source_article` + `entity_mismatches` + окно = ARTICLE_SLICE_CHARS.
- Типо-специфичные промпты + вернуть `item.text` (Show HN/Ask HN) в пайплайн.
- Длина по весу источника вместо фикса 170.

### Стратегически

- Бенч-эксперимент «EN-саммари → перевод на RU» против прямого RU (харнесс готов; на free-моделях это, вероятно, самый большой скачок качества русского).
- Замкнутая петля качества: еженедельный judge-сэмплинг прода + 👍/👎 от читателей + гейт на смену модели.
- Когда появится email: subject = лучший `tldr` дня, секции письма = уже готовые структурированные поля. Вся подготовка выше конвертируется в email-дайджест бесплатно.

## Итог

Каркас (ретраи, guard, бенч) — крепкий, но три системные дыры съедают продукт: **мусор на входе** (нет content-extraction), **ломаный русский на выходе** (маленькая модель без языкового контроля) и **недоинвестированные комментарии** (главная ценность — с единственной попытки, без заголовка и без валидации). Первые два пункта чинятся обвязкой за дни и не требуют лучших моделей; третий — самый большой продуктовый рычаг, потому что пересказ споров HN — единственное, чего нет ни у одного конкурента.
