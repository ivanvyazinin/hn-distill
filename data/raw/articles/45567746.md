I‚Äôve been building [coding tools to help LLMs write q/kdb+ code](/@gabiteodoru/0-6-failure-rate-solving-llm-code-generation-for-q-kdb-4b3ed29f64bd), and one of the quants I spoke to asked: ‚ÄúCould we get it to write really terse code?‚Äù

The q/kdb+ community has a long tradition of preferring terse code, and they feel quite strongly that this is the right way to write code. I used to be in that camp myself before LLM coding assistants came along, as I use small screens and like to minimize keypresses. Nowadays, however, when it‚Äôs LLMs writing my code, all I care about is keeping my LLMs performing well. Because of that, I think the aesthetic preference for terseness should give way to the preference for LLM accuracy, which may mean more verbose code. So to answer that quant‚Äôs question: Yes, we could get it to write really terse code, but I really think we shouldn‚Äôt.

Let‚Äôs start with an example: `(2#x)#1,x#0` is code from the [official q phrasebook](https://code.kx.com/phrases/matrix/#identity-matrix-of-order-x) for constructing an x-by-x identity matrix. It‚Äôs roughly equivalent to Python `np.tile(np.concatenate([[1],x*[0]]),x)[:x*x].reshape(*2*[x])`. If this seems confusing, worry not: I asked Claude to explain it below! ## A Theory for Why This is Happening

Consider prompting an LLM to prove a theorem given some axioms, a task at which today‚Äôs LLMs are quite good. The output would be a string containing the proof, which a deterministic, non-LLM proof-checker could decide whether it‚Äôs correct or not.

Now let‚Äôs think about it from the perspective of [Shannon‚Äôs Information Theory](https://en.wikipedia.org/wiki/Information_theory): before the LLM answers, there is something we don‚Äôt know: the proof; afterwards, our ignorance is dispelled¬π. Therefore we can say we have gained some knowledge, which can be [quantified in bits of surprisal following Shannon‚Äôs theory](https://en.wikipedia.org/wiki/Information_theory#Entropy_of_an_information_source). Since in Shannon‚Äôs theory, surprisal is not objective, but rather it is computed relative to some probability distribution, I‚Äôd suggest using the one distribution we have at hand: the distribution over future tokens provided by the LLM itself¬≤.

I find it fascinating that Shannon studied the mathematical properties of information transmission, yet never applied his theory to the study of [what makes some information useful rather than meaningless](https://amatria.in/Thesis/html-thesis/node171.html)! This makes sense: in the pre-LLM era, information was typically produced by humans, whose probability distributions were inaccessible; we‚Äôve arrived at a unique point in history where objectively-useful strings are now produced by agents that expose their output distributions

Now, let‚Äôs think about taking a proof and trying to rewrite it in fewer steps. The shorter proof would either contain the same information, but somehow compressed (lossless compression), or it might leave out some more obvious steps (lossy compression). Following Shannon‚Äôs Theory, we would quantify the amount of information in the output proof as `I(proof) = -sum log P(token | previous tokens)`, and we would have `I(short proof) <= I(long proof)`, with `<` for lossy compression and `=` for lossless compression. In practice I‚Äôd expect we‚Äôd have both some lossy and some lossless compression, but let‚Äôs focus on the lossless part, I think that‚Äôs the more interesting‚Ä¶

When `I(short proof)=I(long proof)`, per-token average surprisal must be lower for the long proof than for the short proof. But since surprisal for a single token is simply `-log P`, that would mean that, on average, the shorter proof is made out of less probable tokens. Average surprisal-per-token, when exponentiated (a monotonic transform), is also known as [perplexity, a common metric used for LLMs and their inputs and outputs](https://huggingface.co/spaces/evaluate-metric/perplexity). This means that, given equal information content, we would expect the shorter proof to have higher perplexity than the longer proof.

When LLMs output tokens, they first compute the output distribution over all tokens (at every step), and then they use some sort of [pruning strategy (top-P)](https://huggingface.co/spaces/evaluate-metric/perplexity) that basically throws away all but the most likely next tokens. If a particular short proof would require certain less probable tokens that wouldn‚Äôt make it in the top-P when prompted to prove a certain theorem, that would mean that the LLM would be absolutely unable to provide us with that particular proof. Instead, it will simply be forced to output a more verbose proof.¬≥

I‚Äôve been talking about proving theorems so far, but I‚Äôd argue the case of writing code to fulfill tasks is quite similar: we are still prompting the LLM to output a string that we can check for correctness. So let‚Äôs imagine we‚Äôve prompted the LLM to write code. Whether we ask it to write terse code from the start, or ask it to rewrite verbose code more tersely, the same information-theoretic principles apply. The LLM _can_ produce terse code when explicitly asked ‚Äî but I think we shouldn‚Äôt ask it to, because terse code has higher perplexity and is therefore harder for the LLM to work with reliably.

When making the code less verbose, similarly to shortening a proof, we could be dealing with either lossy compression (removing comments, type hints, descriptive variable names, etc.) or lossless compression. But what might lossless compression mean? I‚Äôd argue lossless compression in code corresponds to expressing a combination of multiple, simpler operations, as a single, more complex operation; e.g. incrementing a variable with `i += 1` vs `i = i + 1`. My theory above would suggest that the latter would have lower perplexity.

To test this claim, I measured the actual perplexity using [Qwen 2.5 Coder 7B Instruct](https://huggingface.co/Qwen/Qwen2.5-Coder-7B-Instruct). Prompted with ‚ÄúRespond only with code: How do you increment i by 1 in Python?‚Äù, I compared the two valid outputs: `i += 1` has a perplexity of approximately `38.68`, while `i = i + 1` has a perplexity of approximately `10.88`. The compound operator version packs the same semantic information into fewer tokens, resulting in nearly 4√ó higher perplexity.

This confirms the theoretical prediction: when the same information is compressed into a terser representation, the average surprisal per token increases. And [empirical research has shown](https://arxiv.org/abs/2212.04037v2) that higher input perplexity correlates with lower task performance in LLMs. This suggests that prompting an LLM with terser code than it might have otherwise produced will cause the LLM to systematically underperform at a task like debugging, explaining, or extending it, compared to using the more verbose version of the code that the LLM naturally produces.

This information-theoretic view reveals a broader principle: surprisal per token represents the cognitive throughput the LLM can handle during generation. For straightforward tasks, LLMs naturally produce an appropriate amount of verbosity that keeps per-token surprisal manageable. But for very complex tasks, LLMs can underestimate how many tokens they need ‚Äî they attempt to pack too much information into too few tokens, resulting in high per-token surprisal and increased errors. This framework explains why prompting strategies like ‚Äúthink step by step‚Äù or ‚Äúbe verbose‚Äù improve performance on complex tasks: they force the LLM to spread the cognitive load across more tokens, keeping per-token surprisal within bounds.

As for the practical takeaway for q/kdb+ developers: in the LLM era, let your coding assistant write verbose code. It‚Äôs not your fingers doing the typing anymore, and the performance gains from working with lower-perplexity code might outweigh any aesthetic preference for terseness.

### Footnotes

\[1\] There is an argument to be made, popularized by [Wittgenstein](https://philosophy.stackexchange.com/questions/26494/is-propositions-of-logic-are-tautologies-wittgenstein-literal-or-mystical), that deducing a theorem from axioms shouldn‚Äôt provide us with any additional information, as all information contained in a theorem is already contained in all of the necessary starting axioms, and therefore all theorems are tautologies which add nothing to our knowledge (this is also one way to view the situation from the Bayesian interpretation of Information Theory). Here I‚Äôd counter that what we seek is not whether the theorem is true or false, but a string representing a proof that is validated by a proof-checker, and such a string itself is not contained in the axioms.

\[2\] We‚Äôd ideally want to measure surprisal relative to our own internal distribution (taking a [Bayesian view](https://www.inference.org.uk/mackay/itila/) of our beliefs and ignorance), however, given that our LLM is actually quite good at proving theorems, I‚Äôm happy to assume the distributions are reasonably similar.

\[3\] For completion, I should mention that LLMs can (and do) backtrack if they realize they are going into a dead-end with their initial proof approach. However, since the output so far now impacts the distribution of future tokens, this means that any backtracking will impact the distribution of tokens in the ‚Äúsecond attempt‚Äù. But it also means that my described approach doesn‚Äôt really fit this case, only cases where the LLM outputted the proof without backtracking.
