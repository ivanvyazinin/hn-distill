# Supporting BiDirectional text

[](#supporting-bidirectional-text)

## According to a [new standard](https://terminal-wg.pages.freedesktop.org/bidi/recommendation/)

[](#according-to-a-new-standard)

Please note, that specification is preliminary yet, and [VTE implementation](https://terminal-wg.pages.freedesktop.org/bidi/implementations.html#vte) is still work in progress:

*   all [libvte](http://ftp.gnome.org/pub/GNOME/sources/vte/) based terminals - [https://gitlab.gnome.org/GNOME/vte/issues/53](https://gitlab.gnome.org/GNOME/vte/issues/53)
    *   **libvte**\-based [Gnome Terminal](https://help.gnome.org/users/gnome-terminal/stable/) - [https://bugs.launchpad.net/ubuntu/+source/vte/+bug/263822](https://bugs.launchpad.net/ubuntu/+source/vte/+bug/263822)
    *   **libvte**\-based [sakura](http://www.pleyades.net/david/projects/sakura)
    *   **libvte**\-based [xfce4-terminal](http://docs.xfce.org/apps/terminal/start)
    *   **libvte**\-based [Terminator](http://gnometerminator.blogspot.com/p/introduction.html)
    *   **libvte**\-based [Tilix](https://github.com/gnunn1/tilix) - written in D. Similar user interface as for Terminator.
    *   **libvte**\-based [Lilyterm](http://lilyterm.luna.com.tw/) - [https://github.com/Tetralet/LilyTerm/](https://github.com/Tetralet/LilyTerm/)
    *   **libvte**\-based [ROXTerm](http://roxterm.sourceforge.net/)
    *   **libvte**\-based [evilvte](http://www.calno.com/evilvte/) [https://github.com/caleb-/evilvte](https://github.com/caleb-/evilvte)
    *   **libvte**\-based [Termit](https://github.com/nonstop/termit)

PuTTY (which is more famous in Windows but also runs on linux, known as \`pterm\`) also has usefully good bidi 👍 - \[x\] Aligns RTL lines to the right side, processing them as RTL base-direction text. It seems to be using first-strong heuristic to decide line direction. - \[x\] Treats each line as "paragraph" for purpose of bidi algorithm. - \[ \] Long line wrapping creates separate "paragraphs". This is sub-optimal as a long logical line may get broken into paragraphs of different base direction \[depending on break points\]([https://terminal-wg.pages.freedesktop.org/bidi/prior-work/terminals.html#common-problems](https://terminal-wg.pages.freedesktop.org/bidi/prior-work/terminals.html#common-problems)). AFAICT wrapping is permanent — resizing window to increase width doesn't reflow to join them, they remain broken at original wrap point; resizing to reduce width \_hides\_ text without reflowing — but it's not lost, resizing back reveals it. This mixed model is a bit weird. - \[x\] Supports "paired bracket" addition of newer bidi algorithm, so that e.g. \`HEBREW f(x) TEXT\` looks like \`TXET f(x) WERBEH\` not \`TXET (f(x WERBEH\` as it tended to look once upon the time.

[@wolandark](https://github.com/wolandark)

In any VTE-based terminal, `echo '\e[?2501h'` to enable auto-direction, and you should get the same behavior.

This comes from terminal-wg's [BiDI draft proposal](https://terminal-wg.pages.freedesktop.org/bidi/recommendation/escape-sequences.html#other-output-modes).
