# [Lenses](#Lenses)[](#Lenses "Permalink")

Accessors.jl is build around so called lenses. A Lens allows to access or replace deeply nested parts of complicated objects.

# [Example](#Example)[](#Example "Permalink")

```julia-repl
julia> using Accessors

julia> struct T;a;b; end

julia> obj = T("AA", "BB");

julia> lens = @optic _.a
(@o _.a)

julia> lens(obj)
"AA"

julia> set(obj, lens, 2)
T(2, "BB")

julia> obj # the object was not mutated, instead an updated copy was created
T("AA", "BB")

julia> modify(lowercase, obj, lens)
T("aa", "BB")
```

# [Interface](#Interface)[](#Interface "Permalink")

Implementing lenses is straight forward. They can be of any type and just need to implement the following interface:

*   `Accessors.set(obj, lens, val)`
*   `lens(obj)`

These must be pure functions, that satisfy the three lens laws:

```julia
@assert lens(set(obj, lens, val)) â‰… val
        # You get what you set.
@assert set(obj, lens, lens(obj)) â‰… obj
        # Setting what was already there changes nothing.
@assert set(set(obj, lens, val1), lens, val2) â‰… set(obj, lens, val2)
        # The last set wins.
```

Here `â‰…` is an appropriate notion of equality or an approximation of it. In most contexts this is simply `==`. But in some contexts it might be `===`, `â‰ˆ`, `isequal` or something else instead. For instance `==` does not work in `Float64` context, because `get(set(obj, lens, NaN), lens) == NaN` can never hold. Instead `isequal` or `â‰…(x::Float64, y::Float64) = isequal(x,y) | x â‰ˆ y` are possible alternatives.

See also [`@optic`](../docstrings/#Accessors.@optic-Tuple{Any}), [`set`](../docstrings/#Accessors.set), [`modify`](../docstrings/#Accessors.modify).
