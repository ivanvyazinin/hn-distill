# Wireshark 4.6.0 Supports macOS pktap Metadata (PID, Process Name, etc.)

Published [October 14, 2025](https://nuxx.net/blog/2025/10/)

[![](https://nuxx.net/blog/wp-content/uploads/2025/10/image-1024x502.png)](https://nuxx.net/blog/wp-content/uploads/2025/10/image.png)

Four years after my post on doing [network captures on macOS with Process ID](https://nuxx.net/blog/2021/10/29/network-capture-with-process-name-and-pid-on-macos/), [Wireshark 4.6.0 has been released](https://www.wireshark.org/docs/relnotes/wireshark-4.6.0.html) which includes support for parsing this extra metadata, including the process info.

So how do you do it? Easy! You just need the `pktap` interface parameter.

From the [tcpdump(1) man page](https://manp.gs/mac/1/tcpdump):

> Alternatively, to capture on more than one interface at a time, one may use “pktap” as the _interface_ parameter followed by an optional list of comma separated interface names to include. For example, to capture on the loopback and en0 interface:
> 
> tcpdump -i pktap,lo0,en0
> 
> An _interface_ argument of “all” or “pktap,all” can be used to capture packets from all interfaces, including loopback and tunnel interfaces. A pktap pseudo interface provides for packet metadata using the default PKTAP data link type and files are written in the Pcap-ng file format. The RAW data link type must be used to force to use the legacy **pcap-savefile**(5) file format with a ptkap pseudo interface. Note that captures on a ptkap pseudo interface will not be done in promiscuous mode.

Therefore, we just need something like:

```
tcpdump -i pktap,en0 -w outfile.pcapng
```

or

```
tcptump -i pktap,all host 192.168.0.6 -w outfile.pcapng
```

And then open `outfile.pcapng` in _Wireshark_ and under _Frame_ → _Process Information_ you can find the process name, PID, etc. (See screenshot above.)

Filtering can be done with `frame.darwin.process_info` as listed [here](https://www.wireshark.org/docs/dfref/f/frame.html). For example:

```
frame.darwin.process_info.pname == "firefox"
```

or

```
frame.darwin.process_info.pid == 92046
```

This is super helpful to figure out both what unexpected network traffic is being generated by and the inverse, what a process is doing on the network. And now thanks to Wireshark 4.6.0 it’s even easier.
