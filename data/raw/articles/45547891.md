# Eon

## A programmable nameserver

Published 21 Apr. 2025.

Tags: [research](./research.html "All pages tagged 'research'."), [projects](./projects.html "All pages tagged 'projects'."), [self-hosting](./self-hosting.html "All pages tagged 'self-hosting'.").

[Eon](https://github.com/ryanGibb/eon) is an Effects-based OCaml Nameserver using [MirageOS](https://mirage.io/)’s functionally pure Domain Name System (DNS) [library](https://github.com/mirage/ocaml-dns) with [direct-style IO](https://github.com/ocaml-multicore/eio) (as opposed to [monadic IO](https://en.wikipedia.org/wiki/Monad_\(functional_programming\)#IO_monad_\(Haskell\))) using [OCaml 5](https://ocaml.org/releases/5.0.0)’s [effect handlers](https://ocaml.org/manual/5.0/effects.html) [\[1\]](#ref-sivaramakrishnanRetrofittingEffectHandlers2021) created as the basis to implement some of ideas from the [Spatial Name System](sns.html).

## [§](#dns-tunnelling)DNS Tunnelling

DNS is well-known to be used for [data exfiltration](https://en.wikipedia.org/wiki/Data_exfiltration) and [tunneling](https://en.wikipedia.org/wiki/Tunneling_protocol), since DNS is one of the few protocols that’s almost always allowed through a firewall (at least through a recursive resolver) since it’s so fundamental to the operation of the Internet. I’ve implemented a transport layer over DNS [in Eon](https://github.com/RyanGibb/eon/tree/main/lib/transport), a simple [netcat](https://github.com/RyanGibb/eon/tree/main/bin/transport/netcat) example shows how this can be used to transport data using DNS queries. Many thanks to [Paul-Elliot](https://github.com/panglesd) for helping clean up the transport interface and making it more idiomatically functional. At the moment there’s no multiplexing – a server can only handle one communication at a time, but that could addresses by adding a demultiplexing field (a ‘port’).

The well-defined interfaces that OCaml gives us allows this to be combined in interesting ways, such as a [shell over DNS](https://github.com/RyanGibb/eon/tree/main/bin/transport/sod) (SoD), or an [IP tunnel](https://github.com/RyanGibb/eon/tree/main/bin/transport/tunnel). Note that you wouldn’t want to use this in production without some form of encryption (maybe [ocaml-tls](https://github.com/mirleft/ocaml-tls)?) and authentication (e.g. public/private keys, or capabilities). A standalone example of a capability interface to a shell can be found at [capability-shell](https://github.com/RyanGibb/capability-shell).

There’s some interesting performance characteristics of this tunneling in a variable asymmetry of latency between the sender and receiver, since we’re retrofitting bidirectional packet switching onto a request response protocol. That is, for the DNS server to send data to a client it has to have a query to respond to. We can’t wait to respond to a query until we have data, since recursive resolvers aggressively timeout and return a `SERVFAIL` in the case of a delayed reply. So we have the client periodically poll the server with queries containing no data, so the latency of the server to client link is bounded by the period of this polling.

This is interesting as it allows us to bootstrap communication with a nameserver using nothing but DNS.

## [§](#capability-interface)Capability Interface

DNS is an old protocol, and has some baked-in limitations due to [protocol ossification](https://en.wikipedia.org/wiki/Protocol_ossification) (such as a maximum domain name length of 255 bytes). The ‘back-end’ of the protocol, interactions between services under your control, is easier to evolve. The AXFR zone transfers defined with the Domain Name System [\[2\]](#ref-DomainNamesImplementation1987) are often replaced with some form of database replication in nameserver implementations. Dynamic updates [\[3\]](#ref-vixieDynamicUpdatesDomain1997) using secret key transaction signatures [\[4\]](#ref-eastlake3rdSecretKeyTransaction2000) are often eschewed in favour of custom APIs[1](#fn1). While using these protocols allows an variety of nameserver implementations to interoperate, in practice they are often replaced with custom solutions.

We’ve experimented with a [programmable interface](https://github.com/RyanGibb/eon/blob/main/lib/cap/schema.capnp) to the nameserver with [Cap’n Proto](https://capnproto.org/) [capability](https://en.wikipedia.org/wiki/Capability-based_security)\-[based](http://www.erights.org/elib/capability/ode/index.html) RPCs. This creates capabilities for dynamically updating a domain, or receiving a zonefile and dynamic updates as a secondary nameserver. Please feel free to try deploying it for your own domain, and get [in touch](about.html) if you’d like to set up a reciprocal secondarying relationship.

## [§](#names-have-power)Names Have Power

Having a programmable interface into the domain name system is powerful, because domain names are powerful. Domain names are the root of identity for the Internet protocol suite. Federated communication networks derive user’s identify from domain names including [Matrix](https://matrix.org/), [Mastodon](https://joinmastodon.org/), Bluesky’s AT Protocol [\[5\]](#ref-kleppmannBlueskyProtocolUsable2024), and good old [E-Mail](https://www.rfc-editor.org/rfc/rfc822).

The DNS is also used to prove owership of domains. The security of the modern internet is built on the Transport Layer Security (TLS) protocol [\[6\]](#ref-allenTLSProtocolVersion1999), which uses X509 certificates signed by certificate authorities. The Internet Security Research Group (ISRG)’s Let’s Encrypt certificate authority (CA) provides the [majority](https://w3techs.com/technologies/overview/ssl_certificate) of the Internet’s certificates, over 500 million [in 2025](https://letsencrypt.org/stats/). Traditionally provisioning a certificate was costly and manual process, but the Automatic Certificate Management Environment (ACME) protocol [\[7\]](#ref-barnesAutomaticCertificateManagement2019) used by Let’s Encrypt allows for an automated provisioning of certificates by proving ownership of a domain by displaying a token with one of a number of challenges; HTTP, TLS-ALPN, and DNS.

Only the DNS challenge is possible if the address the domain name points to is not publicly accessible, which is often the case for remote and resource constrained devices behind NATs or firewalls. However, it requires a [complex dance](https://certbot-dns-rfc2136.readthedocs.io/en/stable/) of managing DNS UPDATE keys and specifying the subdomain and zone which it can modify. With our [capability interface](#capability-interface) to the nameserver we can expose fine-grained access control to provision a certificate for a subdomain.

## [§](#wake-on-dns)Wake-on-DNS

Motivated by a desire to curb the power use of self-hosted services which are often idle for large periods of time, such as [storage servers](nas.html), we implemented [hibernia](https://github.com/RyanGibb/eon/tree/main/bin/hibernia) nameserver than can wake a machine up on a name resolution with Eon and a OCaml [Wake-on-LAN](https://en.wikipedia.org/wiki/Wake-on-LAN) [implementation](https://github.com/RyanGibb/ocaml-wake-on-lan). We published this idea as ‘[Carbon-aware Name Resolution](papers.html#carbon-aware-name-resolution)’ in [LOCO2024](https://sicsa.ac.uk/loco/loco2024/).

## [§](#what-next)What next?

I’m looking at extending this interface to support additional functionality for networked services such as storage, identity, and more. [Eilean](eilean.html) is an attempt to parameterise a federated service deployment by a domain name leveraging the NixOS deployment system to do so, but it lacks a runtime component.
