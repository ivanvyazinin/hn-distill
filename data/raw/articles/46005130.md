# [Is C++26 getting destructive move semantics?](/questions/79817124/is-c26-getting-destructive-move-semantics)

[Ask Question](/questions/ask)

Advice

*   [c++](/questions/tagged/c%2b%2b "show questions tagged 'c++'")
*   [c++26](/questions/tagged/c%2b%2b26 "show questions tagged 'c++26'")
*   [relocate](/questions/tagged/relocate "show questions tagged 'relocate'")

Can I express a function that consumes an object? Meaning that its destructor is not run on the moved-from object?

Like the proposed library function [trivially\_locate\_at](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2024/p2786r11.html#core.relocate_function) itself?

```
template <class T>
T* trivially_relocate_at(T* dst, T* src);
```

Naively, if the library authors can, so should I.

Problem: Where is the magic sauce? That function signature does not convey that it effectively destructs an object at `src`, or the reverse problem, that it effectively constructs an object at `dst`.

I suspect the answer is no: The few examples I have found are avoiding it by doing manual memory management with placement-new and std::destroy\_at.

Reason for asking: I would like to [propose](https://github.com/anordal/rustlike/blob/master/proposals/destructive%20move.md) what seems missing: Two new pointer qualifiers to express giving and taking ownership. If you can excuse my reuse of the `new` and `delete` keywords for a moment (it doesn't have to be those):

```
template <class T>
T* trivially_relocate_at(new T* dst, delete T* src);
```

This is not about optimizing C++, but salvaging it: In order to have static lifetime analysis (akin to Rust) in C and/or C++, I see no way around adding an ability to express static ownership transfer.

The specification you cite specifically says "`trivially_relocate` and `trivially_relocate_at` are the primitive entry points into the core magic that tracks and manages object lifetimes" and "The function `trivially_relocate` is a sharp tool that requires compiler magic to implement" so yes, they have magic and no, you cannot write this function yourself.

# It was dropped

[https://herbsutter.com/2025/11/10/trip-report-november-2025-iso-c-standards-meeting-kona-usa/](https://herbsutter.com/2025/11/10/trip-report-november-2025-iso-c-standards-meeting-kona-usa/)

> For trivial relocatability, we found a showstopper bug that the group decided could not be fixed in time for C++26, so the strong consensus was to remove this feature from C++26.

And with this, presumably also replaceability: [https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2024/p2786r7.pdf](https://www.open-std.org/jtc1/sc22/wg21/docs/papers/2024/p2786r7.pdf)

> Replaceability is an important property when we want to transform relocation into assignment or vice versa.

However, the answer would still have been no: As identified in the question and the comment, the proposal basically lacked the "semantics" in "destructive move semantics": It would have added a sharp tool for library authors, but no ability to express ownership transfer. Rather than unlocking rustlike lifetime analysis, it would require manual memory management to use. The feature was half-baked. Better luck next time!
