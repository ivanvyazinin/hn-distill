# Porting an entire library to a different language with a sentence

2025-09-19 • Random Labs Team

case studyresearch

So we've been working on agents for a while. We finally decided we would release what we've been working on. If you're reading this you probably have _some_ idea of what slate is.

For a refresher: Slate is an agent that is designed to handle long and difficult tasks in a highly autonomous way.

In this post we'll be going through the experience of porting an open source library using slate.

The open source repo we've chosen in this example is [Browser Use](https://github.com/browser-use/browser-use/commit/bc178da2337c1d07d071855aed360d0078e91a2c), a project with 70.3k stars built for automating browser interaction with llms.

We have a good amount of testimonials from our users about how good slate is, but this is a real task that shows where coding agents falter.

## The numbers

Slate successfully ported the browser-use project to typescript, excluding all tests and examples, in a couple hours for less than $60.00

The trajectory is 583 tool calls, and used 311 requests.

On average in slate we use ~50,000 input tokens and ~2500 output tokens per request leaving us with about ~15550000 input and ~777500 output tokens.

We ran this version of the harness with sonnet, to showcase what the low-end performance of slate is like. Since it is sonnet, the total input cost was ~$46.65 and output cost was $11.6625 for an approximate total of $58.32

### Task Statistics


This is all I started with, to which the agent asked a clarifying question and I responded with the platform I wanted it to build for:

```
[2025-09-19 20:33:52] Agent: I've analyzed the browser-use project and understand it's a comprehensive Python library that enable...
[2025-09-19 20:34:40] User: Yeah make it 1:1 for nodejs
```

At this point, slate simply has knowledge of the codebase and that I want to port it to typescript. That's it.

Slate by default plans out what it is doing for difficult or long running tasks and will make sure it is aligned with what you, the user, want.

The reason for simply confirming the agent's plan is to showcase that without essentially any outside correction, Slate can make good plans and autonomously execute them for extended periods of time, out of the box.

## 80/20 rule and full autonomy

Slate is designed with the intent of working with an end user, which means slate will do the work, but can choose to rely on you for good opinions.

In an ideal case, and for best results, the user of slate would do some upfront discovery work (using slate) and then have slate one-shot 80% of the work, and then iterate with the user for the remaining 20%.

In this case we opted for pushing slate to be as autonomous as it could be while still interacting with a user.

## Building and Testing Phase

The main issue with this is that it missed 4 files that are present in the original python version. At this point, essentially everything else is an exact 1:1 port, which is what we asked for. As agents tend to do, slate declares victory early, but then proceeds to ask what to do next (test it, run it, build it, etc.).

We respond to this by confirming the options we want it to perform and then slate is in a write-build-verify loop:

```
[2025-09-19 21:07:30] Agent: 🎉 PROJECT COMPLETE! 🎉

I have successfully completed the comprehensive TypeScript port of browser-...
[2025-09-19 21:07:44] User: sure build it and then run it
...
[2025-09-19 21:24:03] User: The goal is to make it so that this ts port runs the exact same way as the original python version
```

Once coding agents enter some sort of feedback, this is the happy path. Slate consistently is able to reach functionally correct code when a clear feedback mechanism is in place.

## Autonomous Problem Solving

After this, slate proceeded to aggressively debug the remaining problems **receiving only confirmation or acknowledgment without any guidance**. All the future steps beyond this are a similar confirmation of options slate has provided as responses.

For example, here is one message where Slate provides options and waits for user input:

> **What's Working Now:**  
> \- Browser class: Complete success - creation, context handling, cleanup  
> \- Controller class: Working perfectly - instantiation and action registry  
> \- Agent class: Constructor fixed, initialization successful, LLM handling improved
> 
> **Current Issue:**  
> We have one remaining structural mismatch in the MessageManager. The Agent expects its history to be an AgentHistoryList, but the MessageManager expects a different history structure with an addMessage method. This is an architectural compatibility issue between the two components.
> 
> **Options to Resolve:**  
> 1\. **Quick fix:** Create a bridge/adapter to make the histories compatible  
> 2\. **Structural fix:** Refactor the MessageManager to work with AgentHistoryList  
> 3\. **Bypass for testing:** Disable the MessageManager temporarily to test other functionality
> 
> Which approach would you prefer to resolve this final compatibility issue?

As you can see, Slate **does not** automatically choose to bypass the tests OR resolve the issue, instead slate aligns with the user in this case. The response to this message was simply:

```
[2025-09-19 21:39:53] User: 2
```

## Verifying success

After continued autonomous debugging and implementation, Slate successfully created a working TypeScript port that would start from a blank page and properly attempt to search by visting google.

Every response to the agent was an approval of what the agent was going to do. However, we chose to not isolate it to see what would happen in a user environment, and to prove that when working with a user the autonomy would be retained. **Good news: It was.**
