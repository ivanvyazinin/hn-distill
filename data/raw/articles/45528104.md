Forget Unit Tests — in the 3D world, only reality matters

This article is about testing… No-no, don’t run away, I know what you’re thinking and why your teeth are already grinding!

My story is about testing the process of importing, processing, and rendering 3D models using Blender. There won’t be another copy-paste praising TDD, drawing testing pyramids, and focusing on unit testing.

In the 3D world, this doesn’t work. I’ll tell you why we had to focus on integration tests, how we built a Golden Dataset from hundreds of real models, and what unexpected bugs Blender and other tools can surprise you with.

TL;DR: if you think a CI pipeline with mocked **bpy** is testing, you should definitely read this.

## Here we go

A brief intro about the service itself and its goals. The service I’m talking about is the core of [3DWay](https://www.3dway.io/), a company that allows users to render and store their models, share them, collect them into galleries, and create their own portfolios.

Users can upload a model in any supported format — .usdz, .fbx, .obj, .stl, .blend, and others. The service places it into a pre-prepared scene, scales it, sets up lighting and camera, and renders a 360-degree video.

_(I wrote about the difficulties of scaling, which requires precise fitting of the model to a complex-shaped bounding collision,_ [_in this article_](/@egorich42/auto-scaling-3d-models-in-blender-a-story-of-bugs-rays-and-pain-0c8a146a8cdb)_)_

This rendering service became a challenge for me: I was used to working with REST or gRPC APIs, familiar backend patterns, but here I had to design a pipeline where “business logic” isn’t an SQL query, but something like camera rotation and correct proxy-mesh recalculation.

I’ll talk about choosing the service architecture and its evolution separately — that’s a topic for another article. In this one, let’s focus on one aspect: how to test all of this.

Why? Because the main runtime of our service isn’t Go or Python code. It’s Blender, running in headless mode, loading scenes, cameras, models, and launching renders.

And Blender’s behavior depends on many things — the engine version, model format, importer _(I’ll tell you about the problems with the built-in USD importer sometime)_, presence/absence of animations, coordinate systems, etc.

All of this requires running a real scene, in real Blender, on real hardware, with subsequent comparison of actual parameters: dimensions, positions, scales, FPS, rotation angles.

That’s exactly why we started building our testing strategy top-down almost from the very beginning: from real results to code architecture, not the other way around.

## Integration is King

Before we get into the implementation details of this testing pipeline, let’s remember why the bottom level of the testing pyramid is unit tests.

Testing architecture is part of application architecture, and almost all modern approaches are based on separating business logic from implementation, connecting them through interfaces. Following the Dependency Inversion principle, instead of tightly coupling to a specific database, storage, or API, we design code through interfaces and ports — and test it in isolation, without running Postgres or Firebase.

This makes unit tests fast, predictable, and cheap in CI/CD. We write “pure logic” without thinking about where it will execute. And integration tests remain the second layer — they’re needed to verify the correctness of assembly and configuration, but usually there are fewer of them, they’re heavier and more expensive to maintain.

**But all of this works only if the implementation isn’t the essence of the product.**

In our rendering service, Blender isn’t an easily replaceable tool, it’s something deeply embedded in the process.

Yes, it’s still replaceable, both entirely and in components, but replacing the model importer or render engine, say from Cycles to custom or UE, requires much more than replacing, for example, a DB adapter from MySQL to MongoDB. Large chunks of business logic can simply fall apart due to the presence or absence of certain capabilities in the editor or renderer.

So you can’t “mock Blender” — you have to make sure the import went correctly, that scaling worked properly, that Cycles didn’t destroy materials, and that the camera position matches the reference. And do this by checking the real results of each operation.

## Everybody lies(especially Blender)

We’re used to the idea that you just rely on an external service — it either works as documented, or it doesn’t work and we handle the error.

However, Blender can behave unpredictably. It can lose the pose or animation skeleton of a model after export through Sketchfab, it can ignore nodes or textures if the export was non-standard, etc.

And you can’t just wrap any unsatisfactory result into an error. You have to fix Blender itself.

Which means the only way to be confident in quality is integration testing.

In our case, the testing pyramid flips: its foundation isn’t units, but Blender running in CI with real scenes and models.

Blender is a tool that’s as flexible and powerful as it is janky, with tons of non-obvious problems. And believe me, it’s very powerful :)

For example, the built-in .usd importer can’t properly read models from Sketchfab (which repacks meshes and skeletons in its own way) and can lose the pose of a Sketchfab model.

Moreover, Blender might not throw an error at all — it just imports “however it worked out”, and you’ll only see this in the final frame.

So we can’t limit ourselves to checking render results, like comparing the final MP4 for expected size, format, video length, and resolution. That’s the final step, but everything that happens in the process also needs to be controlled.

## Enter the Kaiju: Our Golden Dataset”

We need tests to check which node a texture is attached to, which meshes went into the proxy object, whether the animation stack was preserved, how transformations were applied, etc.

And at each step we compare the result with the reference.

And here’s where I want to stop for a moment.

Comparing expected vs real is what any test does, basically. And we rarely think much about “expected”. Most often it’s an obvious and small dataset. If we’re checking, say, a transaction status change in a payment system for users in, let’s say, four different statuses, then expected is simple and clear.

Usually it’s 5–6 different variants, in large systems — a few dozen.

But in 3DWay for integration tests we use a Golden Dataset that is a real Kaiju, it consists of several hundred models + data about their ideal size after scaling, position, camera placement, and other attributes.

Why so many? Shouldn’t 1–2 variants per format be enough? Take a couple dozen models and sort them by size, format, and some features (test processing models with all textures loaded or with missing ones) and that’s it.

But, as I already said, the 3D world is full of nuances and extremely non-obvious details. The same model saved in different editors can have different axes, the same model uploaded and downloaded from Sketchfab will have a changed structure. 3D editors, services, versions of software where the model was created or edited — all of this has huge impact.

For example, how about this — a model can have the framerate of the scene it was created in baked into it, and when importing into another scene, the model will overwrite the framerate with its own. Yes, you understand correctly — opening a .blend scene with 60FPS and importing a model into it using Blender’s default importer, the model itself can drop the FPS to, say, 24 FPS (!!).

So our Golden Dataset is truly large — it contains models with all possible nastiness that the 3D world can offer.

It’s time-consuming, it took a lot of manual work searching, describing, and organizing all test samples, but that’s the price of a bulletproof testing pipeline.

## The Iron Rule: Test on Real Hardware

Another very important point when setting up such a pipeline — remember that hardware is also part of the contract.

It’s common practice to run tests on minimal resources — to avoid overloading infrastructure, save on GPU/CPU, and run units faster.

This is a reasonable approach — if the result doesn’t depend on the execution environment.

But in the case of 3D rendering, this isn’t true. Hardware isn’t just an “executor”, it’s part of the production environment that directly affects the correctness and stability of rendering and even Blender’s behavior (some GPU settings affect Cycles and texture loading).

For example, in our Golden Dataset there’s an industrial boiler model from a power plant, consisting of about 21 million polygons.

At one of the scaling stages, processing such a model requires at least 30GB RAM, otherwise — Out-Of-Memory. If the test environment doesn’t match DEV/PROD, then the result can differ from expected, especially in details that matter to users.

It’s more expensive than lightweight CI nodes, but it gives the main thing: confidence that everything that passed tests will render in prod without surprises.

## Final Thoughts

When your backend is just a wrapper over a REST API or an SQL database, you can easily mock the dependencies and write fast unit tests.

But when your “backend” is Blender, FFmpeg, Unreal Engine, or any other complex runtime — the rules change.

Mocks lie. The real Blender doesn’t behave like your test stub.  
Documentation lies. The official API won’t tell you that importing a .usd file from Sketchfab breaks skeletons.  
Even the tool itself lies. Blender can silently import a model “however it feels like” without throwing a single error.

The only source of truth is a real run, with real data on real hardware.

If you’re building something similar — or have already been through this — I’d love to hear about it.  
Especially cases from gamedev, VFX, or scientific simulation — that’s where the real fun begins.
